# Техническое задание: фронтенд TaskFlow (v2)

**Документ:** ТЗ на пользовательский интерфейс веб-сайта и Mini App
**Проект:** TaskFlow — голосовой ассистент-планировщик для людей с СДВГ
**Версия:** 2.0 (заменяет [[website-spec]])
**Дата:** 25.04.2026

> **Этот документ заменяет** [[website-spec]] версии 1.0. Старая версия — для исторического референса, **не использовать как источник истины**. Если что-то противоречит — приоритет у этого документа.

## 1. Зачем эта версия и что главное изменилось

### 1.1. Корневая ошибка предыдущего ТЗ

Предыдущая версия описывала продукт как **CRM-систему** для управления задачами: главный экран — таблица с фильтрами, канбан с drag-n-drop, ручное управление приоритетами и группами, bulk-actions, расширенные настройки. Это правильный паттерн для **дисциплинированного юзера, который активно управляет своими задачами**. Но это **антипаттерн** для целевой аудитории TaskFlow.

Целевая аудитория — **студенты и фрилансеры с СДВГ или без устойчивого навыка планирования**. Что с ними произойдёт через неделю использования CRM-интерфейса:
- Накопится 80 задач — потому что AI разбирает всё, что они наговорят.
- Половина без дедлайнов и приоритетов — не уточнили в момент записи.
- Канбан-колонка TODO раздуется, IN_PROGRESS останется пустой — никто не помнит туда переносить.
- Фильтры превратятся в новую прокрастинацию: "сейчас наведу порядок в тегах и группах".
- Заходить страшно. Через неделю — забыли про продукт. Это путь Todoist/Notion для людей, которые ими и так не пользуются.

### 1.2. Что меняется концептуально

**Assistant-first, не CRM-first.** Главный принцип: пользователь делает **минимум решений**, AI и система делают максимум. Каждое поле в форме, каждый фильтр, каждая кнопка — повод для пользователя задуматься и **отвалиться**.

**Главная метафора UI:** "система говорит тебе что делать сейчас", а не "система показывает тебе твою свалку задач".

| Было (v1) | Стало (v2) |
|---|---|
| Главный экран сайта = `/tasks` (таблица всех задач) | Главный экран = `/` (focus mode: "что делать сейчас", 1-3 задачи) |
| Канбан на главной странице как основа UX | Канбан на отдельной странице "Все задачи", вторичная навигация |
| Множественные chip-фильтры (группа, тег, приоритет, статус, дедлайн, источник) | Один селектор контекста: "Сейчас" / "Сегодня" / "Эта неделя" / "Всё" |
| Юзер сам ставит приоритеты | AI ставит приоритет всегда, юзер только корректирует если не согласен |
| Юзер создаёт группы вручную через `/groups` | AI создаёт группы автоматически по контексту задачи. Страница `/groups` — для управления уже созданными |
| Команд-палитра, bulk-actions, статистика на 6 графиков | Вырезано или сильно упрощено |
| 5 разделов настроек | 1 экран настроек с 4-5 базовыми пунктами |
| Старые DONE копятся вечно в колонке Done | Автоархив через 7 дней, не показываются в дефолте |

### 1.3. Что НЕ меняется

Бэк готов на 70% и **не переписывается** под этот документ. Все эндпоинты, модель данных, авторизация остаются как есть. Дополнительно к существующим API на дне 7.5 добавляются три эндпоинта (`/tasks/focus`, `/tasks/digest`, `/tasks/quick`) — см. апдейт в [[../roadmap]].

Канбан, табличный список, фильтры — **остаются** в продукте. Они нужны для защиты Web-курсача (drag-n-drop, SSR, фильтры — обязательные технические артефакты). Просто перемещаются на второй уровень навигации, под раздел "Все задачи".

Telegram бот, draft-flow и Mini App — концепция та же, но с упором на голосовой ввод как primary action.

### 1.4. Что вырезано полностью

Список зафиксирован, чтобы агент в IDEA случайно не наделал:
- Command Palette по Cmd+K
- Bulk-actions в таблице (массовый выбор и операции)
- Inline-edit всех колонок таблицы — оставлен только статус
- Расширенная статистика (6 графиков → 1 экран с 3 KPI)
- Множественные chip-фильтры — заменены одним контекстным селектором
- Раздел "Источник создания" в фильтрах
- Settings → Внешний вид → Плотность интерфейса
- Экспорт данных, удаление аккаунта (post-MVP)
- History/audit-лог в детальной карточке (post-MVP)
- Кастомные иконки и цвета групп вручную (AI назначает дефолты)

## 3. Контекст продукта (ужатый)

TaskFlow — голосовой ассистент для людей, у которых **много мыслей и плохо с организацией**. Продукт убирает все формы создания задач: пользователь говорит/пишет в свободной форме, AI разбирает на структурированные задачи и **сам** расставляет приоритеты, дедлайны, группы. Главный экран показывает 1-3 задачи "что делать сейчас", всё остальное скрыто в "Все задачи".

**Целевая аудитория:** студенты 18–25 с СДВГ, фрилансеры с потоком задач, люди без сложившейся привычки планирования. Активно используют Telegram, привыкли к голосовым сообщениям.

**Главная ценность UI:**
1. Activation cost при создании задачи = 1 действие (запись голоса).
2. Activation cost при просмотре "что делать" = 0 действий (открыл — увидел).
3. AI берёт на себя организационные решения. Пользователь решает только "сделать или не сделать".

## 5. Концепция UX: focus mode + secondary navigation

Сайт и Mini App структурно строятся вокруг **двух уровней внимания**:

**Уровень 1 — Focus (главный, 90% времени).** Пользователь видит:
- "Что делать сейчас" — 1-3 задачи.
- Большая голосовая кнопка для добавления.
- Минимум всего остального.

Это default view. Открыл сайт/MiniApp → попал сюда → понял что делать → закрыл.

**Уровень 2 — Manage (вторичный, 10% времени).** Пользователь нажимает "Все задачи" и попадает в полноценный CRM-интерфейс:
- Список всех задач с одним селектором контекста.
- Канбан-доска (для веб-курсача — обязательно).
- Управление группами.
- Статистика.

Это режим "разобраться с накопившимся". Заходим туда раз в день/неделю.

**Принцип:** все CRM-фичи существуют, но они не первое, что видит пользователь. Главное — фокус.

## 6. Карта маршрутов (сайт)

| URL | Доступ | Рендеринг | Уровень | Назначение |
|---|---|---|---|---|
| `/` (если не залогинен) | публичный | SSG | — | Лендинг с кнопкой "Войти через Telegram" |
| `/` (если залогинен) | защищённый | SSR | **Focus** | Главный экран: focus mode |
| `/login` | публичный | CSR | — | Telegram Login Widget |
| `/all` | защищённый | SSR | Manage | Список всех задач + контекстный селектор |
| `/board` | защищённый | CSR | Manage | Канбан-доска (для защиты курсача) |
| `/groups` | защищённый | CSR | Manage | Управление группами (просмотр + переименование) |
| `/stats` | защищённый | SSR | Manage | Один экран статистики (3 KPI + streak) |
| `/settings` | защищённый | CSR | Manage | Простые настройки, 5 пунктов |
| `/task/[id]` | защищённый | SSR | — | Детальная карточка задачи |
| `*` (404) | публичный | static | — | Страница "не найдено" |

Маршруты `/tasks` (старое имя) **переименовано в `/all`** — отражает суть "все задачи" и не путается с focus-режимом на `/`.

### 6.1. Группировка маршрутов в Next.js

```
app/
  (public)/
    page.tsx              ← лендинг (для незалогиненных)
    login/page.tsx
  (authed)/
    layout.tsx            ← общий layout с навигацией
    page.tsx              ← / focus mode (для залогиненных)
    all/page.tsx
    board/page.tsx
    groups/page.tsx
    stats/page.tsx
    settings/page.tsx
    task/[id]/page.tsx
  api/
    auth/telegram/route.ts
    auth/logout/route.ts
    proxy/[...path]/route.ts
  middleware.ts
```

Корневой `/` обслуживается двумя разными `page.tsx` в зависимости от наличия cookie. Реализуется через middleware: если есть `accessToken` → rewrite на `(authed)/page.tsx`, иначе → `(public)/page.tsx`.

## 8. Авторизация (без изменений из v1)

### 8.1. Сайт

1. Открывается `/login` → Telegram Login Widget.
2. Callback → POST на `/api/auth/telegram` → бэкенд `/auth/telegram-login` → JWT в httpOnly cookies.
3. Редирект на `/` (focus mode).

### 8.2. Mini App

1. Telegram передаёт `initData` при открытии.
2. Автоматический POST на `/auth/telegram-miniapp` при старте.
3. JWT в памяти Zustand store, не в localStorage.

### 8.3. Logout

Кнопка в settings → POST `/api/auth/logout` → cookies чистятся → редирект на `/`.

## 9. Дизайн-система

### 9.1. Палитра (без изменений из v1)

CSS-переменные через Tailwind, маппинг под Park UI токены. Темы: светлая (по умолчанию), тёмная. `next-themes`, авто по системной.

Базовые токены: `background`, `foreground`, `card`, `primary`, `secondary`, `muted`, `accent`, `destructive`, `border`.

### 9.2. Цвета приоритетов

- `LOW` → нейтральный серый
- `MEDIUM` → синий
- `HIGH` → оранжевый
- `URGENT` → красный

### 9.3. Цвета статусов

- `TODO` → серый
- `IN_PROGRESS` → синий
- `DONE` → зелёный
- `CANCELLED` → приглушённый

### 9.4. Типографика

- Шрифт интерфейса: **Inter** через `next/font` (subset cyrillic + latin).
- Шрифт моноширинного: **JetBrains Mono**.
- Размеры: дефолтные классы Tailwind.

### 9.5. Spacing, радиусы, тени

- Базовая единица: 4px.
- Радиусы карточек: `rounded-xl` (12px) — крупнее, чем в v1, для более "мягкого" ассистент-вайба.
- Радиусы кнопок: `rounded-lg` (8px).
- Тени: `shadow-sm` в светлой теме на карточках, `shadow-lg` на модалках. В тёмной без теней.

### 9.6. Иконки (Tabler)

- Размеры: 16/20/24/32px.
- Импорт по одной для tree-shaking: `import { IconCheck } from '@tabler/icons-react'`.
- Стиль outline по умолчанию, filled — для активных состояний.

### 9.7. Анимации

- Базовые переходы: `transition-all duration-200`.
- Появление focus-задачи на главной: fade + slight slide-up.
- Завершение задачи (галочка): чекмарк + быстрая анимация исчезновения карточки.
- Запись голоса: пульсирующий радиус + waveform.

### 11.3. Содержание блоков

**Шапка:**
- "Привет, [Имя]" — берётся из Telegram first_name.
- Дата сегодня ("Пятница, 25 апреля") — без года, без времени, кратко.

**Карточки задач (1-3 штуки):**
- Берутся из `GET /api/v1/tasks/focus`.
- На карточке: иконка приоритета, название, дедлайн (если есть, в человеческом формате — "через 4 часа"), группа, опционально оценка времени.
- Две кнопки: **"Сделано"** (primary) и **"Отложить"** (secondary).
- Никаких других действий (редактировать, удалить, посмотреть детали) — на главном экране нет. Хочешь подробнее — на кнопке-стрелочке справа уголком, ведёт на `/task/[id]`.

**Действия с карточкой:**
- "Сделано" → POST `/tasks/{id}/complete` → карточка анимированно исчезает (чекмарк + fade+slide), на её место подгружается следующая (если есть).
- "Отложить" → открывает мини-меню (popover): "На 1 час", "До завтра", "На неделю", "Без даты". Выбор → PATCH с новым deadline → карточка исчезает.

**Под карточками:**
- Строчка "Ещё N задач на этой неделе" с ссылкой на `/all`.
- Если `N == 0` — строчка не показывается.

**Внизу — большая кнопка "Что-то ещё?":**
- Это вход в **quick-input** (см. §12).
- На desktop эта кнопка — дублирует функцию sidebar-кнопок "Записать" / "Написать", но более заметна на focus-экране.

**Состояние "Совсем нет задач" (новый юзер):**

├──────────────────────────────────────────────────┤
│            [✓ Создать всё]    [Отмена]           │
└──────────────────────────────────────────────────┘

### 12.4. Поведение

**Голос:**
1. Пользователь жмёт большую круглую кнопку с микрофоном.
2. Если разрешения нет — браузер просит. Если отказ — fallback на textarea.
3. Запись начинается, идёт волновая визуализация.
4. Лимит 2 минуты, отображается обратный отсчёт после 1:30.
5. Пользователь жмёт "Готово" → POST `/tasks/from-voice` (multipart) → состояние 3.

**Текст:**
1. Пользователь пишет в textarea, жмёт "Разобрать".
2. POST `/tasks/from-text` → состояние 3.

**Подтверждение:**
- Все задачи по дефолту checked (готовы к созданию).
- "✏️ Изменить" на задаче — раскрывает inline-форму с полями (приоритет, дедлайн, группа), которые можно подкрутить.
- "Создать всё" → POST `/tasks/batch-confirm` с id выбранных → закрытие модалки → toast "Создано N задач" → на главной `/` обновляется focus-список.
- Если только одна задача и не нужна правка — есть быстрый путь "Создать" (one-click).

### 12.5. Edit-режим в подтверждении

Карточка задачи разворачивается в форму:
```
┌──────────────────────────────────────────┐
│ ☑ Купить молоко                          │
│ ─────────────────────────                │
│ Заголовок: [Купить молоко         ]      │
│ Приоритет: [Среднее ▼]                   │
│ Дедлайн:   [Завтра 18:00 ▼]              │
│ Группа:    [Покупки ▼]                   │
│ Время:     [10 мин ▼]                    │
│           [Свернуть]                     │
└──────────────────────────────────────────┘
```

Минимум полей — title, priority, deadline, group. **Нет** полей: description, tags, recurrence, estimate (оставляем для детальной страницы).

### 12.6. Mobile

Та же логика, но в формате Drawer снизу на полную высоту экрана. Все размеры и тапаемые цели — минимум 44×44px. Кнопка микрофона на состоянии 1 — 200×200px, на полэкрана.

### 13.3. Контекстный селектор (главный фильтр)

Один Tabs-компонент с 4 вариантами:

- **Эта неделя** — все на ближайшие 7 дней

### 13.4. Поиск

Простой текстовый поиск по `title`. Debounce 300мс. Состояние в URL (`?q=...`).

### 13.5. Колонки таблицы

Только 4 колонки: Задача (название), Приоритет, Дедлайн, Статус. Без чекбоксов, без bulk-actions, без множественных фильтров.

**Inline-edit:** только статус (клик → select). Остальное — открывается детальная карточка.

**Hover на строке:** появляются кнопки "✓ Сделано" и "✏️ Открыть".

### 13.6. Пагинация

50 задач на страницу. Простая нумерация без выбора per-page.

### 13.7. Mobile

Таблица превращается в карточки:
```
┌─────────────────────────────────┐
│ 🔥 Доделать презентацию          │
│ Сегодня 18:00 · Работа           │
│ [✓]                              │
└─────────────────────────────────┘
```

Контекстный селектор — горизонтальный scroll-tabs.

### 13.8. Состояния

- **Загрузка:** Skeleton-строки.
- **Пусто:** "В этом контексте задач нет. Переключи или создай новую."
- **Ошибка:** "Не удалось загрузить. Повторить?"

### 13.9. SSR

Первая страница рендерится на сервере с дефолтным контекстом ("Сегодня"). Дальше — клиентская навигация.

## 20. Создание задач — quick paths

### 20.1. Принцип

В любом месте приложения у пользователя должен быть **прямой путь к голосовому вводу**. Желательно ≤ 1 клика.

### 20.2. Сводка путей

| Откуда | Как |
|---|---|
| `/` (focus) | Кнопка "Что-то ещё?" внизу + sidebar-кнопки |
| `/all` | Кнопка "+ Создать" в шапке + sidebar |
| `/board` | Sidebar |
| Любая страница на mobile | FAB "Записать" в bottom nav |
| Любая страница на desktop | Sidebar-кнопки + горячая клавиша `N` |

### 20.3. Полная форма "вручную"

Полная форма (с description, tags, recurrence, estimate) **скрыта по дефолту**. Доступ:
- В quick-input → "✏️ Изменить" на задаче после AI-разбора → расширенный edit
- На детальной карточке `/task/[id]`
- Никакой кнопки "Создать вручную с полями" в основной навигации нет

## 21. Состояния и обратная связь

### 21.1. Loading

- На `/` — SSR, данные приходят с HTML
- На `/all`, `/board` — Skeleton при загрузке списка
- При мутациях — оптимистичный апдейт, спиннер только в самой кнопке

### 21.2. Empty states

Везде где может быть пусто — спокойный empty state без восклицательных знаков и agressive CTA.

| Где | Текст |
|---|---|
| `/` нет задач | "Всё сделано на сегодня" / "Расскажи, что у тебя на голове" |
| `/all` нет задач в контексте | "В этом контексте задач нет" |
| `/board` пусто в колонке | "Перетащи сюда задачу" |
| `/groups` нет групп | "Группы появятся после первой задачи" |
| `/stats` нет данных | "Покажу когда наберётся неделя данных" |

### 21.3. Error states

- 401 → редирект на `/login` + toast "Сессия истекла"
- 503 (nlp-worker недоступен) → toast "AI временно отдыхает. Попробуй текстом или попозже" + всё равно даём кнопку текстового ввода
- 429 (rate limit) → toast "Слишком быстро. Через минуту попробуй снова" + countdown
- Network error → "Не вижу интернет. Проверь подключение" + retry

### 21.4. Toast-уведомления (Sonner или Park UI Toast)

- Успех мутации: 3 секунды
- Ошибки: 6 секунд + кнопка "Повторить" где уместно
- Отмена недавно сделанного действия: toast с "Отменить" 5 секунд (например, после удаления)

## 22. ⭐ Адаптивность (без изменений из v1, повторяю как critical)

> **Блокирующее требование для приёмки.** Сайт должен работать одинаково отлично на телефоне и на ПК.

### 22.1. Брейкпоинты

| Брейкпоинт | Ширина | Tailwind |
|---|---|---|
| Mobile S | 320–639px | default |
| Mobile L | 640–767px | sm: |
| Tablet | 768–1023px | md: |
| Desktop | 1024–1279px | lg: |
| Desktop L | 1280–1535px | xl: |
| Desktop XL | ≥1536px | 2xl: |

### 22.2. Принципы

- **Mobile-first** в Tailwind-классах
- **Не скрывать функциональность на мобиле** — менять UX, не выкидывать
- **Тапаемые цели ≥ 44×44px**
- **Шрифт основного текста ≥ 14px на мобиле, 16px желательно**
- **Никаких горизонтальных скроллов** кроме намеренных (канбан, графики)

### 22.3. Тестовые ширины

320 / 375 / 390 / 412 / 768 / 1024 / 1280 / 1440 / 1920 / 2560.

### 22.4. Ключевые трансформации

| Элемент | Desktop | Tablet | Mobile |
|---|---|---|---|
| `/` focus | sidebar + main | свёрнутый sidebar | bottom nav |
| `/all` таблица | 4-колонки | те же | карточный список |
| `/board` | 3 колонки рядом | 2 видны | snap-scroll по 1 |
| Quick-input | Modal по центру | Modal | Drawer снизу на полный экран |
| Кнопка записи в quick-input | 200×200px | 180×180px | 220×220px (главное на мобиле) |
| Все edit-popover'ы | popover | popover | Drawer |

## 23. Производительность

### 23.1. Цели

- **LCP**: ≤ 2 сек на `/` (это критично — focus-экран должен быть мгновенным), ≤ 2.5 сек на лендинге, ≤ 3 сек на `/all`
- **INP** ≤ 100мс
- **CLS** ≤ 0.1
- **Lighthouse Desktop**: Performance ≥ 90 на лендинге и `/`, ≥ 80 на остальном
- **Lighthouse Mobile**: Performance ≥ 80 на лендинге и `/`, ≥ 70 на остальном

### 23.2. Техники

- Code splitting на уровне маршрутов (Next.js auto)
- Динамические импорты для тяжёлого: VoiceRecorder, Recharts, DateTimePicker
- Next/Image с sizes
- next/font для Inter и JetBrains Mono с display: swap
- Tabler Icons — именованные импорты
- TanStack Query: stale 30s для focus, 5min для групп

## 26. Критерии приёмки

### 26.1. Функциональные

- [ ] `/` показывает 1-3 focus-задачи и работает мгновенно (SSR)
- [ ] Создание задач голосом проходит end-to-end (запись → AI → подтверждение → создание)
- [ ] Создание задач текстом работает
- [ ] Контекстный селектор на `/all` фильтрует корректно (Сейчас/Сегодня/Эта неделя/Всё)
- [ ] Drag-n-drop канбана работает с оптимистичными апдейтами
- [ ] Inline-edit статуса в `/all` работает
- [ ] AI автоматически назначает группы и приоритеты при разборе
- [ ] Авторизация через Telegram Login Widget работает
- [ ] Mini App открывается в Telegram, авторизуется автоматически
- [ ] Простые настройки сохраняются автосейвом
- [ ] Streak в статистике считается корректно

### 26.2. Технические

- [ ] Next.js 14 App Router
- [ ] SSR на `/`, `/all`, `/task/[id]`, `/stats`
- [ ] SSG на лендинге
- [ ] CSR на `/board`, `/groups`, `/settings`
- [ ] TypeScript strict, типы из OpenAPI
- [ ] Park UI как библиотека компонентов
- [ ] Tabler Icons как иконки
- [ ] Все формы валидируются Zod
- [ ] Адаптивность по всем критериям §22
- [ ] Тёмная и светлая темы
- [ ] Lighthouse Desktop: Performance ≥ 90 (лендинг и `/`), Accessibility ≥ 90
- [ ] Lighthouse Mobile: Performance ≥ 80 (лендинг и `/`), Accessibility ≥ 90

### 26.3. UX

- [ ] Открытие приложения → видно "что делать" за ≤ 2 секунды
- [ ] Создание задачи голосом ≤ 10 секунд от начала записи до карточки в focus
- [ ] На пустом focus — спокойное empty state, не "Создай задачу!!!"
- [ ] Никаких бесконечных свалок задач на главной
- [ ] Mobile FAB "Записать" — primary action

### 26.4. Защита курсачей

Для **Web-курсача** обязательно есть:
- SSR (`/`, `/all`, `/task/[id]`, `/stats`)
- SSG (лендинг)
- CSR (`/board`, `/groups`, `/settings`)
- Drag-n-drop (`/board`)
- Адаптивность (демо переключение десктоп → мобильник)
- Голосовой ввод через Web Audio API (`MediaRecorder`)
- Тёмная/светлая тема
- Работа с REST API через JWT proxy

Для **Java-курсача** на стороне сайта дополнительной демо-функциональности не требуется.

Связано с [[../roadmap]] — апдейтом от 25.04.2026.

Manage-режим: `/all` с контекстным селектором и таблицей, `/board` с канбаном и drag-n-drop. Это основа для защиты Web-курсача.
