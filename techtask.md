## 1. Наименование и область применения

**Наименование:** TaskFlow Core Service — серверная часть цифрового ассистента-планировщика.

**Область применения:** личное планирование задач для студентов и специалистов с интенсивным потоком разнородных задач. Система работает через Telegram (бот и Mini App) и браузер.

## 3. Назначение разработки

**Функциональное назначение:** автоматизировать создание структурированных задач из произвольного текстового или голосового ввода с помощью LLM; хранить задачи, управлять их жизненным циклом и доставлять напоминания через Telegram.

**Эксплуатационное назначение:** REST API-сервер, работающий локально в Docker Compose, доступный через ngrok-туннель для интеграции с Telegram Bot API.

## 4. Требования к программе

### 4.1 Функциональные требования

#### Авторизация
- Аутентификация через Telegram Mini App (HMAC-SHA256 верификация initData).
- Аутентификация через Telegram Login Widget.
- Выдача JWT access token (TTL 60 минут) и refresh token (TTL 30 дней, хранится в Redis).
- Ротация refresh token при обновлении; отзыв токена при logout.

#### Управление задачами
- Создание задачи вручную (POST /api/v1/tasks) и через быстрый ввод (POST /api/v1/tasks/quick).
- NLP-разбор произвольного текста (POST /api/v1/tasks/parse-text): извлечение названия, дедлайна, приоритета, категории.
- Черновой режим: задача создаётся с флагом `is_draft=true`, активируется после подтверждения пользователем.
- Полный CRUD: чтение, обновление полей, мягкое удаление (`is_deleted=true`).
- Режим фокуса (GET /api/v1/tasks/focus): возвращает 1–3 задачи с ближайшим дедлайном и наибольшим приоритетом.
- Дайджест на дату (GET /api/v1/tasks/digest).
- Фильтрация по статусу, приоритету, группе.

#### Статусы задачи
`TODO` → `IN_PROGRESS` → `DONE` / `CANCELLED`

#### Приоритеты
`URGENT` > `HIGH` > `MEDIUM` > `LOW`

#### Группы и теги
- CRUD групп (POST/GET/DELETE /api/v1/groups).
- Привязка тегов к задаче (many-to-many).

#### Telegram-интеграция
- Приём webhook-обновлений (POST /telegram/webhook).
- Верификация запросов по заголовку `X-Telegram-Bot-Api-Secret-Token`.
- Идемпотентная обработка: повторная доставка одного `update_id` не создаёт дубликатов (Redis, TTL 1 час).
- Обработка текстовых сообщений: NLP-разбор → черновик → inline-кнопки «Подтвердить» / «Отменить».
- Обработка голосовых сообщений: скачивание OGG → STT через Whisper → NLP-разбор → черновик.
- Команды: `/start`, `/help`, `/today`.

#### Напоминания
- При создании задачи с дедлайном создаётся запись в `scheduled_notifications`.
- Время напоминания: дедлайн минус значение из настроек пользователя (по умолчанию 60 минут).
- notification-worker опрашивает таблицу каждую минуту (FOR UPDATE SKIP LOCKED для параллельного запуска).
- При ошибке: retry с экспоненциальной задержкой, максимум 5 попыток; далее статус `FAILED`.
- При изменении дедлайна старое напоминание отменяется, создаётся новое.

#### Аудит
- Все изменения задачи (создание, обновление, удаление, смена статуса) записываются в `task_events` с типом события и JSONB-payload.

### 4.2 Нефункциональные требования

#### Производительность
- Ответ на стандартный REST-запрос (CRUD): не более 200 мс при нагрузке до 50 одновременных пользователей.
- Обработка NLP-запроса (включая вызов LLM): не более 8 секунд.
- Размер JAR-артефакта после сборки: не более 80 МБ.

#### Надёжность
- Идемпотентная обработка Telegram-обновлений.
- Retry + Circuit Breaker (Resilience4j) при сбоях LLM-провайдера; при открытом circuit breaker — fallback-ответ.
- Корректный запуск всех компонентов после `docker compose restart`.

#### Безопасность
- JWT-подпись: HS256, секрет из переменной окружения.
- HMAC-SHA256 верификация каждого initData и каждого webhook-запроса.
- Пароли и секреты — только через `.env`-файл, не в коде.
- Soft delete: данные пользователя не удаляются физически.

#### Документация
- REST API документируется через springdoc-openapi; Swagger UI доступен по `/swagger-ui.html` при запуске.
- JavaDoc на все публичные методы бизнес-логики (пакеты `application`, `domain`).

### 4.3 Интерфейсы

#### Внешние API
| Сервис | Протокол | Назначение |
|--------|----------|------------|
| Telegram Bot API | HTTPS/REST | Webhook, отправка сообщений, скачивание файлов |
| Groq API | HTTPS/REST | LLM (Llama 3) и STT (Whisper) |
| YandexGPT API | HTTPS/REST | Резервный LLM-провайдер |

#### Внутренние интерфейсы
| Компонент | Протокол | Назначение |
|-----------|----------|------------|
| nlp-worker | HTTP/REST | NLP-разбор текста и аудио |
| PostgreSQL | JDBC | Основное хранилище данных |
| Redis | TCP (Lettuce) | Refresh-токены, идемпотентность webhook |

#### Пользовательский интерфейс
REST API потребляется двумя клиентами: Telegram-бот (webhook) и React Mini App. Прямого UI в core-service нет, кроме Swagger UI для разработки.

## 5. Требования к надёжности

- MTTR (среднее время восстановления после сбоя): не более 2 минут при перезапуске через `docker compose restart`.
- Потеря данных при штатном перезапуске: недопустима (данные в PostgreSQL, токены в Redis с persistence).
- При недоступности Groq API: circuit breaker переходит в открытое состояние после 3 последовательных ошибок; запросы к parse-text возвращают HTTP 503 с понятным сообщением об ошибке.

## 8. Порядок контроля и приёмки

Приёмка производится путём:

1. **Запуска системы** командой `docker compose up -d --build` — все контейнеры должны перейти в статус `healthy`.
2. **Проверки API** через Swagger UI (`/swagger-ui.html`): выполнить авторизацию, создать задачу вручную и через parse-text, получить список задач и фокус.
3. **Проверки Telegram-бота**: отправить текстовое и голосовое сообщение, убедиться что черновики создаются и подтверждаются.
4. **Проверки unit-тестов**: `./gradlew test` — все тесты должны пройти.
5. **Проверки документации**: JavaDoc генерируется без ошибок командой `./gradlew javadoc`.
